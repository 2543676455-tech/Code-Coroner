==================================================
一、项目目标
==================================================

RepoJudge 是一个用于分析 GitHub 开源仓库的 AI Agent。

用户输入一个公开 GitHub 仓库地址后，系统自动：

1. 克隆目标仓库。
2. 读取 README、项目目录、依赖文件和配置文件。
3. 提取 README 中的主要功能声明。
4. 在代码中寻找支持这些声明的证据。
5. 检查测试、Docker、环境变量、日志、异常处理、CI/CD 和安全风险。
6. 在受限 Docker 沙箱中选择性执行测试。
7. 输出带证据的项目审计报告。
8. 生成 README 可信度、生产可用度、学习价值和套壳指数。
9. 支持“专业审计模式”和“毒舌点评模式”。

系统的核心原则：

每一个结论都必须有代码、配置文件、测试结果或运行日志作为证据。
禁止让大模型无依据评分。
评分必须由确定性规则为主、LLM 语义分析为辅。

四、Agent 工作流
==================================================

START
  ↓

  ↓
clone_repository
  ↓

  ↓

  ↓

  ↓

  ↓

  ↓

  ├── 否 → calculate_scores
  └── 是 → sandbox_test → calculate_scores
  ↓

  ↓

定义清晰的 RepoJudgeState，至少包含：

- engineering_checks

- 设置最大重试次数，避免 Agent 死循环

==================================================
五、README 声明核验
==================================================

从 README 中提取可以被验证的声明，例如：

- 支持一键部署
- 支持 Docker
- 支持多 Agent
- 支持生产环境
- 支持身份认证
- 支持流式输出
- 支持向量检索
- 包含完整测试
- 支持某数据库
- 支持异步任务

每个声明转换成结构化对象：

{
  "id": "claim_001",
  "claim": "支持 Docker 一键部署",
  "category": "deployment",
  "verification_rules": [
    "存在 Dockerfile",
    "存在 docker-compose.yml",
    "存在启动命令",
    "配置中不存在明显缺失"
  ],
  "status": "verified | partial | unsupported | unknown",
  "confidence": 0.0,
  "evidence": [],
  "reason": ""
}

核验流程：

1. 先使用规则和代码搜索寻找证据。
2. 再把有限的相关内容交给 LLM 判断。
3. 不要把整个仓库直接塞给模型。
4. LLM 只能解释证据，不能虚构证据。
5. 没有证据时必须输出 unsupported 或 unknown。

证据对象格式：

{
  "type": "file | code | config | test | log",
  "path": "Dockerfile",
  "line_start": 1,
  "line_end": 20,
  "excerpt": "...",
  "description": "项目包含容器启动配置"
}

对 excerpt 做长度限制，避免返回大量源码。

九、评分体系
==================================================

评分不能完全交给 LLM。

实现独立的 deterministic scoring engine。

评分范围均为 0～100。

1. README 可信度

根据：

- README 声明有多少得到代码支持
- 启动文档是否完整
- README 与实际目录是否一致
- 示例命令是否存在对应入口
- 是否存在夸大但无证据的声明

2. 生产可用度

根据：

- 测试
- 部署
- 配置管理
- 日志
- 异常处理
- 健康检查
- CI
- 类型检查
- 安全风险
- 依赖管理

3. 学习价值

根据：

- 代码结构清晰程度
- 文档质量
- 模块化程度
- 是否包含测试
- 是否具有代表性的技术实现
- 是否方便本地运行

4. 套壳指数

套壳指数越高，表示项目越可能只是简单封装已有 API。

参考因素：

- 核心业务代码量
- 是否只有一个模型 API 调用
- 是否存在真实工具调用
- 是否有数据处理流程
- 是否有状态管理
- 是否有评估体系
- README 宣传复杂度与实际实现复杂度的差异

将每项评分公式和权重写入独立配置文件：

backend/app/core/scoring_rules.py

LLM 只能生成不超过总分 10% 的语义辅助分。

报告必须展示主要加分项和扣分项。

十一、API 设计
==================================================

实现以下接口：

POST /api/v1/audits

请求：

{
  "repository_url": "https://github.com/owner/repo",
  "mode": "professional",
  "run_tests": false
}

返回 task_id。

GET /api/v1/audits/{task_id}

返回任务状态：

- pending
- running
- completed
- failed

GET /api/v1/audits/{task_id}/report

返回完整 JSON 报告。

GET /api/v1/audits/{task_id}/report/markdown

返回 Markdown 报告。

GET /health

返回健康检查结果。

首版任务可以使用 FastAPI BackgroundTasks 或受控线程池，不需要引入 Celery。

但需要限制：

- 同时执行的任务数量
- 单仓库最大体积
- 克隆超时
- 分析超时
- 单文件最大读取大小

防止用户提交：

- file://
- 内网 URL
- 本地路径
- 非 GitHub 仓库 URL
- 带命令注入内容的 URL
